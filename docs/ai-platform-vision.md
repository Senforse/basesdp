# SDP · 面向 AI 的工业软件开发平台 · 愿景与共识

> 本文档沉淀「基于 SDP 构建面向未来接入 AI 能力的工业软件开发平台」这一方向的战略共识、关键决策与待定项。
> 它是讨论产物，不是最终路线图；落地里程碑与任务拆解见后续 `docs/engineering-roadmap.md` 及专项规划。
> 形成日期：2026-06。

## 1. 战略主线（核心判断）

**不追求「超大平台先行」。** 工业软件的护城河不是大而全，而是在某个工业场景里做到不可替代。

采用「**长出来**」而非「**设计出来**」的路径：

```
做深一个离散制造场景  →  在它上面叠 A 档 AI  →  数据攒够再上 B 档 AI  →  共性自然沉淀成平台
   (真实业务价值)         (加速开发与使用)        (预测/判别)            (对外可复用的开发平台)
```

平台能力是从第一个做深的场景里沉淀出来的副产品，而不是起点。

## 2. 能力分层（从下往上生长）

1. **工程地基层**：测试网、CI/CD、数据库迁移、可观测性。没有它，上层全是空中楼阁。**这是上 AI 的准入门槛，不是可选项。**
2. **平台能力层**（已有雏形）：代码生成器、RBAC、多租户、统一契约、三类日志。目标是打磨成「元数据驱动」，成为 AI 可介入的接口。
3. **AI 赋能开发层（A 档）**：自然语言 → 数据模型 → 全栈模块；AI 辅助审计与排错；基于审计日志/业务数据的智能问答（RAG）。
4. **AI 赋能业务层（B 档 / C 档）**：缺陷检测、设备故障预测、能耗优化（B 档，需数据）；智能排产、闭环控制（C 档，现阶段不碰）。

正确推进顺序：1 → 2 → 3 → 4，不可跳级。

## 3. 已达成的共识

### 3.1 行业与场景
- 行业：**离散制造**（工单、BOM、工序、装配、质量追溯）。
- 首场景推荐：**工单/生产执行打底 + 质检/质量追溯产出 AI 燃料**。
  - 工单/生产执行是脊柱：纯软件、零硬件依赖、是一切的根、摆脱供应商痛度最高。
  - 质量追溯挂在工序上，每天产出**带标注的结构化缺陷数据**——正是 B 档模型最稀缺的燃料。
  - 二者在同一场景闭环，不是两个孤立项目。
- 暂缓：设备物联/运维（依赖 PLC/传感器硬件，门槛高）、智能排产（依赖全量数据）。

### 3.2 AI 能力顺序
- **A 档先行且贯穿全程**：把 LLM 接在现有代码生成器前面，自然语言描述 → 生成全栈模块。风险低、价值看得见、加速建场景。
- **B 档等数据**：现在没有工单流转/质检缺陷数据，模型无米下锅。场景跑起来积累数据后再切入。
- **C 档现阶段不碰**：决策/控制类责任与安全风险过大。
- AI 在工业里的铁律：**可解释 + 可追溯 + 人在回路**。审计日志（before/after + traceId）是天然优势，AI 的每个建议都要能回溯依据。

### 3.3 私有化部署（必须私有化，数据不出网）
- **推理服务分阶段**：Ollama 起步验证（单机/单人、零采购）→ vLLM 上量（多人并发、OpenAI 兼容、迁移近乎无痛）。
- **模型选型**（许可证友好，可商用、无供应商绑定）：
  - 代码生成/审查：Qwen3-Coder 系（14B 起，32B 更佳）。
  - 智能问答/RAG：Qwen3（思考/非思考双模，8B~14B）。
  - 复杂推理/排错：DeepSeek 系（R1/V4 蒸馏，14B~32B）。
  - 向量化 Embedding：bge-m3 / bge-large-zh 或 Qwen3-Embedding（中文检索专用，必配）。
  - 模型迭代快，采购落地时挑当时最新稳定版，选型逻辑不变。
- **RAG 地基**：独立部署 Milvus 或 Qdrant + bge-m3，**不赌达梦向量类型**（国产化兼容性待验证）。知识源 = 审计日志 + 业务数据 + 工艺文档。
- **Java 接入**：用 Spring AI 连接 Ollama/vLLM 的 OpenAI 兼容端点，团队技术栈不分裂。
- **硬件分档投入**：
  - 起步验证：1×24GB 卡（如 4090），跑 32B 4-bit(AWQ ~20-24GB) 或 14B。
  - 团队共享：1~2×48GB（L20/A100 级），vLLM 跑 32B AWQ，撑 10+ 并发。
  - B 档训练：更强算力，可后置（按需租或采购）。
  - 显存口径：7B/8B 4-bit ≈ 6-8GB；14B 4-bit ≈ 12GB；32B 4-bit ≈ 20-24GB；FP16 翻 2-4 倍。
- 避坑：不要一上来买大显卡赌未验证的方向；不要把向量检索硬塞进达梦。

### 3.4 工程地基（P0 准入门槛）
引用 `docs/engineering-roadmap.md` 的 P0 三件套，作为上 AI 之前必须补齐的门槛：
- 自动化测试脚手架（后端 JUnit5+Mockito、前端 Vitest）。
- CI/CD 门禁（GitHub Actions：mvn verify + 前端 build/test + ensure-utf8 校验）。
- 数据库迁移（Flyway 达梦方言需先做兼容性 PoC，或 Liquibase）。

## 4. 待拍板事项（下次继续）

| 待定项 | 说明 | 影响 |
|---|---|---|
| 首场景最终锁定 | 工单/生产执行 + 质检 的组合是推荐项，尚未最终确认 | 决定第一批要建的数据模型与页面 |
| 信创/国产化合规 | 公司是否受信创约束（央企/涉密/党政相关），需找 IT/合规确认 | 决定 GPU 用 NVIDIA+vLLM 还是华为昇腾+CANN/MindIE，颠覆硬件与推理栈选型 |
| 第一批真实用户 | 车间班组长 / 设备工程师 / 内部 IT 团队 | 决定产品形态与交互优先级 |

## 5. 技术选型补充判断

- **微服务化别急**：业务模块尚空，过早拆分只增运维复杂度。先在 modular monolith 里把第一个场景做深，`sdp-modules` 已为将来拆分预留，需要时再拆。
- **达梦的现实约束**：国产化数据库，Flyway、向量扩展等生态工具兼容性需逐个验证，遇兼容性问题优先选独立成熟组件。
